# Proposal: 消费记录 UI 优化 + OCR 提取精度提升

## Intent

上一版本已实现基础的拍照上传、OCR 识别和消费记录管理。但在使用中发现三个核心问题：
1. **OCR 提取不准** — 正则模式太局限，对实际小票的商户/金额/日期提取率低
2. **记录列表信息模糊** — 无法直观看出哪些文件已上传、哪些缺失
3. **详情页无法续传文件** — 上传结账单后，支付记录和发票缺少便捷的上传入口

同时需要一个**手动打包导出**功能，满足用户将月结文件压缩发送到邮箱或分享的需求。

## Scope

### In Scope
- OCR 正则提取模式全面升级（商户名/金额/日期）
- PaddleOCR 替换方案调研（后续备选）
- 首页记录列表：红色/橘色/绿色卡片，反映文件齐全程度
- 详情页重构：3 个文件区域（结账单/支付记录/发票）+ 缩略图 + 加号上传/替换
- 手动打包导出按钮：ZIP 压缩 → 下载分享 / 发送邮箱

### Out of Scope
- OCR 引擎替换为 PaddleOCR（依赖 pub.dev 网络可达性，留待后续）
- 文件拖拽排序
- 多选批量操作

## Approach

所有改动均在现有 Flutter 纯客户端架构内完成，不需要后端：
- OCR: 保留 ML Kit，升级正则提取模式；备忘 paddle_ocr_flutter 方案
- UI: 基于现有 Card + ListTile 结构改造，新增 InkWell 点击区域
- 打包: 复用现有 `FileService.zipMonthRecords()` 方法，新增 UI 入口

## Key Design Decisions

1. **红/橘/绿三色编码** — 不依赖 RecordStatus 枚举值，而是直接判断 receiptImg/paymentImg/invoicePdf 三个字段的 null 状态。红=全缺，橘=部分有，绿=三证齐全
2. **详情页三个独立文件区域** — 每个区域独立管理自己的上传/替换逻辑，互不影响
3. **加号=替换语义** — 已有文件时加号按钮覆盖在缩略图右上角，点击替换；无文件时显示大的加号区域
4. **打包入口在首页** — 在月份切换和状态卡片下方，用户可随时手动触发
